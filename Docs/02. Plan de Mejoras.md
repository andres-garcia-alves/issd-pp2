## ✅ Plan de Mejoras Priorizado

### 🔴 Prioridad Crítica - Fix crítico (5 minutos): **COMPLETADO**

**Mejora 0 - Restaurar `n_iter=40` en RandomizedSearchCV**

- Cambiar `n_iter=3` a `n_iter=40` (o mínimo 20) en la celda de finetuning del Modelo 1.
- El comentario `#40` en el código confirma que fue reducido solo para debug.
- Dejar en 3 equivale a explorar el espacio de hiperparámetros casi por azar.
- **Acción:** Ejecutar con `n_iter=40` una vez (puede tardar varios minutos) y guardar los resultados en MLflow. Los artefactos del modelo guardados en `Models/` se actualizan automáticamente.

---

### 🟠 Prioridad Alta - Quick wins metodológicos (15-20 min cada uno)

**Mejora 1.1 - Curva Precision-Recall por clase**

- Agregar gráfico de curvas PR para las 3 clases (LEVE / GRAVE / MORTAL) con `precision_recall_curve` + `average_precision_score`.
- Es la métrica correcta para evaluar modelos con clases desbalanceadas; complementa la matriz de confusión actual.
- Se ubica en la sección de evaluación del Modelo 1, después de la matriz de confusión.
- Logear el AP (Average Precision) por clase en MLflow junto a las métricas existentes.

**Mejora 1.2 - Permutation Importance**

- Agregar `permutation_importance` de scikit-learn junto a la `feature_importance` del RF actual.
- La importancia por permutación es más confiable que la impurity-based (no se sesga por features de alta cardinalidad).
- Se muestra en un barplot comparativo lado a lado con el actual, o como gráfico nuevo debajo.
- No requiere reentrenar el modelo.

**Mejora 1.3 - Threshold optimization para MORTAL**

- En lugar de `argmax(predict_proba)`, bajar el umbral de clasificación para la clase MORTAL (ej: 0.25-0.30).
- Objetivo: subir el recall de MORTAL a costa de algo de precisión - apropiado para el dominio (falso negativo en mortal es más costoso que falso positivo).
- Mostrar el trade-off precision/recall con una tabla o curva para justificar el umbral elegido.

---

### 🟡 Prioridad Media - Mejoras de impacto técnico (1-2 horas cada una)

**Mejora 2.1 - XGBoost como modelo comparativo**

- Agregar `XGBClassifier` como Modelo 1-B, entrenado con las mismas features y partición que el RF.
- Usar `compute_sample_weight('balanced', y_train)` para manejar el desbalance (XGBoost multiclase no tiene `class_weight` directo).
- Logear el experimento en MLflow con el mismo esquema de métricas (F1 macro, balanced accuracy, recall por clase).
- El valor principal es la **comparación visual en MLflow** entre RF y XGBoost, que muestra progresión metodológica.
- Actualizar la Demo de Gradio para usar el modelo ganador.

**Mejora 2.2 - SMOTE para mejorar recall en GRAVE/MORTAL**

- Aplicar `SMOTE` sobre `(X_train, y_train)` para generar muestras sintéticas de las clases minoritarias.
- **Importante:** SMOTE se aplica solo sobre el set de entrenamiento; `X_test/y_test` permanecen con distribución real.
- Combinar con `class_weight='balanced_subsample'` ya existente para potenciar el efecto.
- Comparar métricas antes/después de SMOTE en MLflow. Si no mejora, documentarlo: la honestidad metodológica es valorada.

---

### 🔵 Prioridad Baja - Solidez del Modelo 2 (K-Means)

**Mejora 3.1 - Comparar K-Means con un algoritmo alternativo**

- Probar al menos un algoritmo adicional: `HDBSCAN` (no requiere especificar K, detecta clusters de densidad variable) o `GaussianMixture` (clusters elípticos, más flexible que K-Means esférico).
- Comparar ambos con la misma métrica de Silhouette e interpretabilidad de los perfiles resultantes.
- Si K-Means sigue siendo el mejor, la comparación lo justifica. Si el alternativo gana, se adopta.
- El k=2 con Silhouette bajo que ya se menciona en las conclusiones se convierte en una fortaleza metodológica si está acompañado de esta comparación.

---

### 🟢 Diferenciador - Diferenciador para defensa oral

**Mejora 4.1 - SHAP values para interpretabilidad**

- Agregar `shap.TreeExplainer` sobre el modelo final (RF o XGBoost ganador).
- Incluir dos visualizaciones: `summary_plot` global (features más influyentes) y un `waterfall_plot` para un caso individual.
- Integrar en la Demo de Gradio: cuando se hace una predicción, mostrar los top-3 factores que la determinaron (ej: "clasificado como MORTAL principalmente por: tipo_via=autopista, franja=madrugada, modo=MOTO").
- Esta mejora tiene el mayor impacto en la **defensa oral** - permite responder "¿por qué el modelo dice eso?" con evidencia concreta.
